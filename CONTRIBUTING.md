# コントリビューションガイド

Pi-Gottchi への改善提案、バグ報告、ドキュメント修正、コード変更を歓迎します。

## Issue を作成する前に

- 既存の Issue に同じ内容がないか確認してください。
- バグの場合は、再現手順、期待した動作、実際の動作を記載してください。
- Raspberry Pi のモデル、Raspberry Pi OS、Whisplay ドライバー、接続した周辺機器を記載してください。
- ログから API キー、会話内容、ユーザー名、ホスト名などを除去してください。

安全上の問題や秘密情報の漏えいは公開 Issue に書かず、[セキュリティポリシー](SECURITY.md)に従ってください。

## 開発手順

1. リポジトリを fork します。
2. 変更内容ごとにブランチを作成します。
3. 小さく、目的が分かる単位で変更します。
4. 実機またはプレビューで確認します。
5. 変更理由と確認方法を添えて Pull Request を作成します。

```bash
git switch -c fix/short-description
```

## 確認方法

Python ファイルの最低限の構文確認:

```bash
python3 -m compileall app preview.py gen_boot_frames.py
```

音声状態、端末コマンド、サーボ制御の回帰テスト:

```bash
python3 -m unittest discover -s tests -v
```

実機なしで顔の描画を確認:

```bash
python3 preview.py
```

ハードウェアや音声処理を変更した場合は、確認に使った Raspberry Pi と周辺機器、実行した操作、ログの要点を Pull Request に記載してください。

## コーディング上の注意

- Raspberry Pi Zero W の限られたCPU・メモリを考慮してください。
- センサーやオプション依存がなくても、基本機能が起動できる状態を保ってください。
- ネットワークやデバイスの一時的な失敗でメインループを停止させないでください。
- APIキー、`memory.json`、`state.json`、生成音声をコミットしないでください。
- 利用者向けの動作が変わる場合は関連ドキュメントも更新してください。

## Pull Request の説明に含めるもの

- 何を変更したか
- なぜ必要か
- どのように確認したか
- ハードウェア配線や設定への影響
- 互換性や既知の制限

プロジェクトにライセンスが追加されるまでは、第三者によるコードの複製・改変・再配布条件が明確ではありません。コントリビューションを受け付ける前に、所有者がライセンス方針を決めることを推奨します。
